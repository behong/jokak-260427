# Total10Shop 서버 이전 / Docker 전환 정리

작성일: 2026-07-04

## 1. 작업 목적

기존 Windows 서비스 기반으로 운영 중이던 Total10Shop 서비스를 Docker 기반 운영 구조로 전환했다.

기존 운영 방식은 다음과 같았다.

```text
Windows 서비스 / WinSW 기반 Python 서비스
Windows PostgreSQL
Cloudflared 터널
collector-api
publisher-api
collector-worker
media-worker
publisher-worker
```

전환 후 운영 방식은 다음과 같다.

```text
Docker Desktop + WSL2 기반 컨테이너 운영
Windows PostgreSQL 유지
Cloudflared 유지
collector-api Docker 전환
publisher-api Docker 전환
collector-worker Docker 전환
media-worker Docker 전환
publisher-worker Docker 전환
```

## 4. 실제 Docker로 전환된 Windows 서비스

아래 5개는 Docker로 대체되었다.

```text
collector-api
publisher-api
collector-worker
media-worker
publisher-worker
```

현재는 삭제하지 않고 보존 상태로 둔다.


## 7. Docker 테스트 과정 요약

진행한 검증 순서:

```text
1. WSL2 Ubuntu 설치 완료
2. Docker Desktop 설치 및 WSL 연동 완료
3. 테스트 PostgreSQL 컨테이너 생성
4. 운영 DB dump를 테스트 DB에 복원
5. collector-api 테스트 컨테이너 실행 성공
6. publisher-api 테스트 컨테이너 실행 성공
7. 외부 발행 토큰 비활성화 확인
8. worker 이미지 빌드 성공
9. collector-worker --once 성공
10. media-worker --once 성공
11. publisher-worker --once 성공
12. shadow API로 운영 DB 연결 리허설 성공
13. worker 3개 Docker 전환 성공
14. publisher-api Docker 전환 성공
15. collector-api Docker 전환 성공
16. Cloudflared 외부 도메인 health 성공
```

## 8. 운영 DB 연결 방식

현재 Docker 컨테이너는 Windows PostgreSQL에 아래 방식으로 접속한다.

```text
host.docker.internal:5432
```

DB URL은 `.env.docker.cutover`에 저장되어 있다.

주의:

```text
.env.docker.cutover는 실제 DB 비밀번호를 포함하므로 절대 Git에 커밋하지 않는다.
```

DB 비밀번호에 특수문자 `#`가 포함되어 있었으므로 URL에는 인코딩된 형태로 들어가야 한다.

## 10. Docker 운영 재시작 명령

전체 Docker 운영 서비스 재시작:

```bash
docker compose -f docker-compose.cutover.local.yml --profile workers restart
```

전체 Docker 운영 서비스 실행:

```bash
docker compose -f docker-compose.cutover.local.yml --profile workers up -d
```

전체 Docker 운영 서비스 중지:

```bash
docker compose -f docker-compose.cutover.local.yml --profile workers stop
```

주의:

```bash
docker compose -f docker-compose.cutover.local.yml --profile workers down
```

위 명령은 컨테이너를 제거한다. 롤백이나 재구성이 필요할 때만 사용한다.

## 12. 안정화 후 Windows 서비스 처리 계획

즉시 삭제하지 않는다.

권장 단계:

### 1일차

상태 유지:

```text
Stopped + Manual
```

문제 없으면 Disabled로 변경 가능:

```bash
powershell.exe -NoProfile -Command "Set-Service collector-api -StartupType Disabled"
powershell.exe -NoProfile -Command "Set-Service publisher-api -StartupType Disabled"
powershell.exe -NoProfile -Command "Set-Service collector-worker -StartupType Disabled"
powershell.exe -NoProfile -Command "Set-Service media-worker -StartupType Disabled"
powershell.exe -NoProfile -Command "Set-Service publisher-worker -StartupType Disabled"
```

### 3~7일차

문제 없으면 Windows 서비스 등록만 삭제 가능.

삭제 전 최종 백업 필요.

삭제 후보:

```text
collector-api
publisher-api
collector-worker
media-worker
publisher-worker
```

삭제 명령 예시:

```bash
powershell.exe -NoProfile -Command "sc.exe delete collector-api"
powershell.exe -NoProfile -Command "sc.exe delete publisher-api"
powershell.exe -NoProfile -Command "sc.exe delete collector-worker"
powershell.exe -NoProfile -Command "sc.exe delete media-worker"
powershell.exe -NoProfile -Command "sc.exe delete publisher-worker"
```

단, 서비스 등록만 삭제한다. 파일은 바로 삭제하지 않는다.

### 1~2주차

문제 없으면 기존 파일 백업 후 정리 가능:

```text
C:\srv\services
```

단, PostgreSQL과 Cloudflared 관련 파일은 삭제 금지.

## 13. 재부팅 대비 주의사항

Docker Desktop 기반 운영이므로 재부팅 후 Docker Engine과 컨테이너 자동 복구가 중요하다.

확인 명령:

```bash
powershell.exe -NoProfile -Command "Get-Service com.docker.service | Select-Object Name,Status,StartType"
```

Docker Desktop이 자동으로 뜨지 않으면 컨테이너가 자동 복구되지 않을 수 있다.

전환 직후 바로 재부팅하지 말고, 최소 몇 시간 운영 안정화 후 재부팅 테스트를 진행한다.
